# Mothmark UniversalEditor Metadata, Theme, Variant, and Processor Design

## Status

Proposed feature-complete design spec.

This document consolidates the UniversalEditor ideas discussed so far and expands them into a coherent implementation plan for Mothmark. It is written as a design and engineering handoff document, not as final production code.

## Purpose

Mothmark needs a UniversalEditor that can reshape its UI from a Zod schema while still allowing each editor surface to feel designed rather than autogenerated. The editor should support room data, world data, author-defined commands, conditions, effects, aliases, entity references, flags, and future schema-backed data without requiring a new hand-built form every time.

The core idea is:

> The Zod schema defines valid data. Editor metadata defines how that data should feel to edit. Processors define smart behavior around that data. Themes and variants define how controls look and behave in different editor contexts.

This gives Mothmark a universal editing engine that is generic by default, specialized by registration, and still aligned with the product's quiet, tactile, night-archive design direction.

---

## Relevant project context

### Canonical data model

The Mothmark spec establishes that JSON is the canonical save format. The map, inspector panel, and code view are different representations of the same underlying JSON data. This is important for UniversalEditor because the editor should mutate JSON data directly rather than generate intermediate text and parse it back.

UniversalEditor should therefore operate on:

- a Zod schema
- a JSON-compatible value
- editor metadata
- an editing context
- a controlled `onChange` or patch callback

It should not know about database persistence, save checkpoints, raw syntax parsing, or runtime execution. Those systems can wrap it.

### Author-defined command model

The author command language spec requires commands to be UI-editable, serializable into world JSON, inspectable, safe, and free of arbitrary JavaScript or TypeScript. Authored commands are made from structured rule data:

- trigger phrases
- aliases
- connector patterns
- target requirements
- conditions
- effects
- messages
- priority
- fallback behavior
- scope
- resolution behavior
- branches

UniversalEditor should make this kind of nested schema practical to edit without exposing raw object structure as the main experience.

### Design direction

Mothmark should feel like a practical authoring tool over an old archive desk: room records, map labels, command transcripts, field notes, warning marks, index cards, and manuscript-like editing surfaces. The editor must not become generic SaaS UI with a surface-level fantasy skin.

The design doc also emphasizes:

- real author workflows before atmosphere
- clear state and plain controls
- concrete action labels
- visible validation
- keyboard accessibility
- readable contrast
- no color-only meaning
- consistency of terms like room, exit, command, checkpoint, warning, and transcript

UniversalEditor metadata should support those goals directly.

# 1. Design goals

## 1.1 Universal but not bland

UniversalEditor should be able to render any supported schema tree, but important Mothmark concepts should get custom renderers and processors.

Generic examples:

- string input
- number input
- boolean toggle
- enum select
- nested object
- array of collapsible items

Specialized Mothmark examples:

- condition builder
- effect list
- entity picker
- flag picker
- alias processor
- connector processor
- command preview
- branch summary
- pattern summary

## 1.2 Schema-driven, metadata-shaped

The schema should define the data contract. Metadata should define how the editor presents and organizes that data.

The same schema should be able to produce different editing experiences by swapping metadata themes and variants.

Example:

```ts
<UniversalEditor
  schema={AuthorCommandSchema}
  value={command}
  onChange={setCommand}
  meta={AuthorCommandEditorMeta}
/>
```

The same `AuthorCommandSchema` might look like:

- a friendly authoring form in `archive` theme
- a dense command rules workbench in `command-workbench` theme
- a developer inspection panel in `developer` theme

## 1.3 Every data type is represented

UniversalEditor should support every data shape currently used or expected in the Mothmark schemas:

- strings
- long strings
- player-facing messages
- numbers
- booleans
- enums
- arrays
- objects
- optional fields
- defaulted fields
- literals
- unions
- discriminated unions
- recursive structures
- condition groups
- effect arrays
- entity references
- flag keys
- read-only previews
- hidden/internal fields

## 1.4 No empty chrome

Every control type should support optional `title` and `description` in metadata.

Rendering rule:

- If `title` exists, render it.
- If `description` exists, render it.
- If neither exists, render only the control.
- Do not reserve empty label space.
- Do not render empty wrapper elements just because a field could have a title.

This allows dense layouts and polished layouts to use the same renderer system.

## 1.5 Metadata coverage

Every editable object field should have metadata, either explicit or generated by a known metadata helper.

Raw schema fields like this should eventually fail a metadata coverage test:

```ts
name: z.string();
```

Fields should be declared through helpers or annotated with metadata:

```ts
name: editorString({
  title: "Name",
  description: "Human-readable editor name.",
});
```

or:

```ts
name: z.string()
  .min(1)
  .meta({
    editor: {
      control: "text",
      title: "Name",
      description: "Human-readable editor name.",
    },
  });
```

Exception: generated metadata can be allowed for small structural fields such as discriminators and literals when wrapped by a higher-level discriminated union helper.

# 2. Core architecture

## 2.1 High-level pipeline

UniversalEditor should be built as a renderer pipeline:

```txt
UniversalEditor
  -> SchemaIntrospector
  -> MetadataResolver
  -> RendererResolver
  -> FieldRenderer
  -> ProcessorRegistry
  -> Patch/Change System
```

## 2.2 UniversalEditor responsibilities

UniversalEditor owns:

- accepting schema, value, metadata, and context
- introspecting the schema into a normalized schema tree
- resolving metadata
- selecting renderers
- applying path-based changes
- running processors at the right lifecycle points
- displaying validation and editor warnings
- honoring title/description visibility rules
- passing accessibility information to controls

UniversalEditor should not own:

- game save logic
- server mutations
- raw syntax parsing
- command runtime execution
- hardcoded world-specific behavior
- custom business rules that belong in processors

## 2.3 SchemaIntrospector

The SchemaIntrospector converts Zod internals into a normalized `SchemaNode` tree so renderer components do not need to understand Zod internals directly.

Conceptual shape:

```ts
type SchemaNode =
  | StringNode
  | NumberNode
  | BooleanNode
  | EnumNode
  | LiteralNode
  | ObjectNode
  | ArrayNode
  | UnionNode
  | DiscriminatedUnionNode
  | OptionalNode
  | NullableNode
  | DefaultNode
  | UnknownNode;
```

Each node should carry:

```ts
type BaseSchemaNode = {
  kind: string;
  path: EditorPath;
  schema: z.ZodTypeAny;
  metadata: ResolvedEditorFieldMeta;
  description?: string;
  isOptional?: boolean;
  hasDefault?: boolean;
  defaultValue?: unknown;
};
```

## 2.4 MetadataResolver

Metadata can come from multiple places:

1. Inline Zod metadata.
2. Zod registries.
3. External metadata maps.
4. Theme defaults.
5. Local overrides passed into UniversalEditor.
6. Inferred fallback metadata.

Recommended precedence:

```txt
field-level override
-> external metadata map
-> inline schema metadata
-> registered schema metadata
-> theme default for control
-> universal inferred default
```

This lets schema authors define stable metadata once while screen-specific editors can override grouping, visibility, density, or variants.

## 2.5 RendererResolver

RendererResolver chooses a component for a field.

Priority:

1. Explicit `renderer` metadata, if supported.
2. Explicit `control` metadata.
3. Registered custom renderer for schema or metadata tag.
4. Inferred renderer from schema node kind.
5. Unsupported-field fallback.

A field with `control: "condition-builder"` should render the custom condition builder even if its underlying schema is a recursive object/discriminated union.

## 2.6 ProcessorRegistry

Processors provide smart behavior around generic data.

A processor can:

- normalize a value
- validate a value
- provide option lists
- generate summaries
- create default values
- duplicate values safely
- resolve references
- generate previews
- compute warnings
- suggest fixes

Conceptual API:

```ts
type EditorProcessor<TValue = unknown, TContext = UniversalEditorContext> = {
  name: string;

  normalize?: (value: TValue, context: TContext) => TValue;

  validate?: (value: TValue, context: TContext) => EditorIssue[];

  getOptions?: (context: TContext) => EditorOption[];

  getSummary?: (value: TValue, context: TContext) => string;

  createDefault?: (context: TContext) => TValue;

  duplicate?: (value: TValue, context: TContext) => TValue;

  preview?: (value: TValue, context: TContext) => string;
};
```

## 2.7 Patch/change system

Nested editors should use path-based patches rather than passing many nested setters.

Recommended form:

```ts
type EditorPath = Array<string | number>;

type EditorPatch = {
  path: EditorPath;
  value: unknown;
};
```

UniversalEditor can expose:

```ts
type UniversalEditorProps<TValue> = {
  schema: z.ZodType<TValue>;
  value: TValue;
  onChange: (nextValue: TValue) => void;
  onPatch?: (patch: EditorPatch) => void;
  meta?: EditorObjectMeta;
  context: UniversalEditorContext;
};
```

Internally, field renderers can call:

```ts
updateAtPath(path, nextValue);
```

This makes deeply nested arrays and objects manageable.

# 3. Universal editor context

UniversalEditor needs world/editor context so fields can offer useful options.

```ts
type UniversalEditorContext = {
  world?: World;
  selectedRoomId?: string;
  selectedEntityId?: string;
  selectedCommandId?: string;

  mode?: "author" | "developer" | "debug";

  registries: {
    processors: ProcessorRegistry;
    renderers: RendererRegistry;
    optionSources: OptionSourceRegistry;
  };

  getRoomOptions?: () => EditorOption[];
  getItemOptions?: () => EditorOption[];
  getFeatureOptions?: (roomId?: string) => EditorOption[];
  getNpcOptions?: () => EditorOption[];
  getTagOptions?: (kind?: string) => EditorOption[];
  getFlagOptions?: () => EditorOption[];
  getTopicOptions?: () => EditorOption[];
  getDirectionOptions?: () => EditorOption[];
};
```

Entity pickers, flag pickers, condition builders, and effect builders should depend on this context rather than hardcoding access to global state.

# 4. Metadata model

## 4.1 Base metadata

Every control type should support this shared metadata:

```ts
type BaseEditorFieldMeta<TControl extends EditorControlType> = {
  control: TControl;

  title?: string;
  description?: string;

  variant?: string;
  theme?: EditorThemeName;
  density?: EditorDensity;
  mood?: EditorMood;

  hidden?: boolean;
  readOnly?: boolean;
  disabled?: boolean;
  advanced?: boolean;
  deprecated?: boolean;

  group?: string;
  section?: string;
  order?: number;

  layout?: EditorLayoutMeta;

  processor?: string;
  processors?: string[];

  issues?: EditorIssue[];
};
```

`title` and `description` are optional for every field. They should be hidden when not provided.

## 4.2 Layout metadata

```ts
type EditorLayoutMeta = {
  width?: "full" | "half" | "third" | "quarter" | "compact";
  inline?: boolean;
  indentation?: "none" | "normal" | "deep";
  emphasis?: "normal" | "quiet" | "advanced" | "danger" | "success";
  stickyHeader?: boolean;
  startCollapsed?: boolean;
};
```

Layout should affect placement and hierarchy, not validation or saved data.

## 4.3 Density and mood

Do not encode every visual choice into themes. Add independent knobs:

```ts
type EditorDensity = "compact" | "comfortable" | "spacious";
type EditorMood = "plain" | "quiet" | "dramatic" | "technical";
```

This lets Mothmark support combinations like:

- `archive + compact`
- `archive + spacious`
- `developer + comfortable`
- `scriptorium + quiet`

without duplicating themes.

## 4.4 Control-specific discriminated metadata

For stronger TypeScript support, define control-specific metadata as a discriminated union:

```ts
type EditorFieldMeta =
  | TextFieldMeta
  | TextareaFieldMeta
  | MessageFieldMeta
  | NumberFieldMeta
  | ToggleFieldMeta
  | SelectFieldMeta
  | MultiSelectFieldMeta
  | TagListFieldMeta
  | StringListFieldMeta
  | ObjectFieldMeta
  | ArrayFieldMeta
  | DiscriminatedUnionFieldMeta
  | ConditionBuilderFieldMeta
  | EffectListFieldMeta
  | EntityPickerFieldMeta
  | FlagPickerFieldMeta
  | CodePreviewFieldMeta
  | HiddenFieldMeta;
```

Each field meta type extends the base:

```ts
type TextFieldMeta = BaseEditorFieldMeta<"text"> & {
  features?: TextControlFeatures;
};
```

This is cleaner than having one large `features` object containing optional keys for every control.

## 5.1 Default schema-to-control inference

When metadata does not explicitly set a control, infer one:

| Zod shape                            | Default control       |
| ------------------------------------ | --------------------- |
| `z.string()`                         | `text`                |
| `z.string()` with long-text metadata | `textarea`            |
| player-facing message fields         | `message`             |
| `z.number()`                         | `number`              |
| `z.boolean()`                        | `toggle`              |
| `z.enum([...])`                      | `select`              |
| `z.array(z.string())`                | `string-list`         |
| `z.array(z.object(...))`             | `array`               |
| `z.object(...)`                      | `object`              |
| `z.discriminatedUnion(...)`          | `discriminated-union` |
| known `ConditionSchema`              | `condition-builder`   |
| known `EffectSchema[]`               | `effect-list`         |
| known entity id fields               | `entity-picker`       |
| known flag fields                    | `flag-picker`         |

# 7. Variants by control type

Control describes the interaction. Variant describes the visual and UX treatment.

Example:

```ts
{
  control: "string-list",
  variant: "alias-chips",
  theme: "command-workbench",
  title: "Aliases",
}
```

## 7.1 `text`

Short single-value text input.

Variants:

| Variant            | Use                                                  |
| ------------------ | ---------------------------------------------------- |
| `plain-field`      | Default standard input                               |
| `field-note`       | Tactile, label-heavy field for names and short notes |
| `inline-label`     | Compact row-style field                              |
| `stamp-field`      | Good for IDs, stable keys, generated identifiers     |
| `searchlike-field` | Good for text that behaves like lookup/filter input  |

Recommended uses:

- `name`: `field-note`
- `id`: `stamp-field`
- quick filters: `searchlike-field`
- dense object rows: `inline-label`

Features:

```ts
type TextControlFeatures = {
  placeholder?: string;
  prefix?: string;
  suffix?: string;
  maxLength?: number;
  copyable?: boolean;
  clearable?: boolean;
  generateButton?: boolean;
  normalize?: "none" | "trim" | "slug" | "lowercase" | "id";
  autocomplete?: string;
};
```

# 10. Discriminated unions

Discriminated unions are central to authored commands and should receive dedicated treatment.

## 10.1 Required behavior

A discriminated union renderer should:

1. Detect the discriminator key.
2. Show a friendly selector for the variant.
3. Show variant labels and descriptions from metadata.
4. Render only the selected variant fields.
5. Create valid default data when changing variant.
6. Preserve compatible/shared fields when configured.
7. Confirm before destructive variant switches when configured.
8. Keep the discriminator visible only when helpful.

## 10.2 Variant switching

When switching variants, the renderer should ask the selected variant schema or processor to create default data.

Example:

```ts
type VariantSwitchOptions = {
  preserveSharedFields: boolean;
  resetOnVariantChange: boolean;
};
```

If switching from:

```json
{"type": "room", "roomId": "kitchen"}
```

to:

```json
{"type": "room-tag", "tag": "safe"}
```

the renderer should not leave stale `roomId` in the value unless the schema allows it and preservation is explicitly intended.

## 10.3 Creation mode

For arrays of discriminated union items, creation can use a `choice-cards` or `wizard-step` variant even if existing items use `rubric-tabs` or `compact-switcher`.

Example:

- Add pattern -> choose from cards like `Verb only`, `Verb + target`, `Speech`, `Direction`.
- Existing pattern card -> use compact variant switcher.

# 16. Processors

Processors keep UniversalEditor generic while making specific fields smart.

## 16.1 Required initial processors

### `aliases`

For alias lists and verbs.

Responsibilities:

- trim whitespace
- lowercase where appropriate
- dedupe
- preserve multi-word aliases
- sort longest-first when matching behavior requires it
- warn on empty strings
- warn on duplicates after normalization

### `connectors`

For command connector words.

Responsibilities:

- trim
- lowercase
- dedupe
- preserve multi-word connectors like `out of`
- sort longest-first before parser use
- warn when connector list is empty and pattern requires connectors

### `unique-id`

For generated ids.

Responsibilities:

- generate stable IDs
- avoid collisions against existing objects with `.id`
- support prefixes like `branch`, `effect`, `command`, `pattern`

### `command-pattern-summary`

Generates readable pattern summaries.

Examples:

```txt
verb only: listen / wait
verb + target: examine apple
verb + object + connector + target: put apple on table
speech: ask cook about rats
raw phrase: open sesame
```

# 17. Metadata helper functions

Metadata should be easy to apply consistently. Avoid writing large `.meta({ editor: ... })` blocks everywhere.

Recommended helpers:

```ts

editorMessage();
editorBoolean();
editorNumber();
editorEnum();
editorArray();
editorObject();
editorDiscriminatedUnion();
editorCondition();
editorEffects();
```

Example:

```ts
export const AuthorCommandSchema = z.object({
  id: editorString({
    title: "Command ID",
    description: "Stable unique id for this authored command.",
    variant: "stamp-field",
    features: {
      normalize: "id",
      copyable: true,
      generateButton: true,
    },

    title: "Name",
    description: "Human-readable editor name for this authored command.",

```

```ts
withEditorMeta(z.string().min(1), {
  control: "text",
  title: "Name",
});
```

# 18. Inline metadata vs external metadata

Support both.

## 18.1 Inline metadata

Best for stable field identity and control type.

Example:

```ts
name: z.string()
  .min(1)
  .meta({
    editor: {
      control: "text",
      title: "Name",
    },
  });
```

## 18.2 External metadata maps

Best for screen-specific layout, grouping, and variants.

Example:

```ts
const AuthorCommandEditorMeta = {
  theme: "command-workbench",
  groups: [
    {id: "basics", title: "Basics", order: 10},
    {id: "matching", title: "Matching", order: 20},
    {id: "behavior", title: "Behavior", order: 30},
  ],
  fields: {
    description: {
      group: "basics",
      variant: "compact-area",
    },
    effects: {
      group: "behavior",
      variant: "action-stack",
    },
  },
};
```

## 18.3 Recommended split

Use inline metadata for:

- control type
- title
- description
- basic feature behavior
- processor identity

Use external metadata for:

- grouping
- ordering
- theme
- density
- advanced visibility
- screen-specific variant override

# 19. Validation and issues

UniversalEditor should display both schema validation issues and editor-specific issues.

## 19.1 Issue model

```ts
type EditorIssueSeverity = "info" | "warning" | "error";

type EditorIssue = {
  severity: EditorIssueSeverity;
  path: EditorPath;
  message: string;
  source: "schema" | "processor" | "reference" | "editor";
  fix?: EditorIssueFix;
};

type EditorIssueFix = {
  label: string;
  apply: () => void;
};
```

## 19.2 Issue display rules

- Errors should be shown near the field and summarized at the group/editor level.
- Warnings should be visible but not panic-styled.
- Info should be quiet.
- Do not rely on color alone.
- Icons need labels or accessible text.
- Collapsed objects/arrays should show validation badges.

## 19.3 Reference validation

Processors should detect broken references:

- missing room id
- missing item id
- missing feature id
- missing NPC id
- unknown flag usage, when not allowed
- invalid entity type for target requirement
- stale branch/effect IDs

# 20. Accessibility requirements

UniversalEditor should be built with accessibility as part of the baseline, not a later pass.

## 20.1 General requirements

- Every interactive control needs an accessible name.
- Metadata `title` can be used as the visible label when present.
- If no title is present, the parent renderer must still provide an accessible name.
- Description text should be connected with `aria-describedby` when useful.
- Keyboard navigation must work for all controls.
- Focus indicators must be visible.
- Collapsible sections need correct expanded/collapsed state.
- Reorder controls need keyboard-accessible alternatives.
- Add/remove/duplicate buttons need explicit labels.
- Error messages must be connected to fields.
- Color cannot be the only status indicator.

## 20.2 Pattern-specific guidance

Use WAI-ARIA APG patterns for:

- accordion/disclosure behavior
- combobox/search picker behavior
- listbox behavior
- tabs behavior
- menu button behavior, if used
- tree behavior, if the condition builder uses tree semantics

Prefer native HTML controls where they satisfy the UX.

## 20.3 Mothmark-specific accessibility stance

The night-archive style must not reduce readability. Atmosphere should be created through spacing, borders, labels, texture, and restrained color, not low contrast text.

# 26. Final design principle

Build UniversalEditor around this rule:

> Generic by default, specialized by registration.

The generic renderer should make schema-backed editing possible immediately. Registries, processors, custom renderers, themes, variants, and metadata should make important Mothmark workflows feel intentional.

That keeps the system powerful without making the editor hardcoded to one schema or one visual style.
